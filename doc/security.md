# Security Measures

## Authentication

Admin access uses `POST /login` and an httpOnly session cookie. Credentials are passed via environment variables (`ADMIN_USER`, `ADMIN_PASS`). The login page is CSRF-protected and renders a real token in a hidden field. Sessions are stored in Redis for multi‑replica support.

## Session Management

- **Inactivity timeout**: Sessions expire after **15 minutes** of inactivity (`rolling: true`).
- **Session ID regeneration**: The session ID is regenerated on successful login to prevent session fixation attacks.
- **Secret rotation**: The `SESSION_SECRET` is rotated daily. Redis stores the active secret list, with the current secret plus up to seven prior daily secrets retained for 7 days before pruning.

## CSRF Protection

The admin dashboard uses **CSRF tokens** to protect state-changing endpoints. The token is generated by the `csrf-csrf` library (double‑submit cookie pattern) and embedded in the HTML page as a meta tag and a global `window.csrfToken` variable.

All `fetch` requests from the admin dashboard include the token in the `CSRF-Token` header (or `X-CSRF-Token`). Requests without a valid token are rejected with a `403` status. The `/csrf-token` endpoint exists for browser flows and tests. The CSRF protection is applied only to admin routes that change state.

## Dashboard Rendering

Values rendered in the dashboard are escaped before being inserted into HTML or inline JavaScript to reduce XSS exposure.

## Double Anonymity

Client IP addresses are never logged or stored; only an irreversible SHA256 hash with a daily rotating salt is kept. Rate limiting uses the hashed IP.

## IP Allowlisting (Optional)

For clients with strict security requirements, you can restrict which IP addresses or CIDR ranges can call the `/api/v1/verify` endpoint using a given API key. When an allowlist is set, requests from other IPs receive a `403 Forbidden`.

To set or update the allowlist, use the admin endpoint `PATCH /api/v1/keys/:api_key/ip-allowlist` with a JSON body `{ "allowed_ips": ["192.168.1.0/24", "10.0.0.5"] }`.

The allowlist is stored in the `api_keys` table as a PostgreSQL `TEXT[]` array. Both exact IP addresses and CIDR notation (e.g., `192.168.1.0/24`) are supported.

## HTTPS

In production, the service should be deployed behind a reverse proxy with HTTPS enabled. Set `SESSION_SECRET` to a strong random value. Security headers (Helmet) and CORS are configured conditionally based on the `PUBLIC_URL` protocol.
