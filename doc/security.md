# Security Measures

## Authentication

Admin access uses `POST /login` and an httpOnly session cookie. Credentials are passed via environment variables (`ADMIN_USER`, `ADMIN_PASS`). The login page is CSRF-protected and renders a real token in a hidden field.

## CSRF Protection

The admin dashboard uses **CSRF tokens** to protect state-changing endpoints, including webhook management. The token is generated by the `csurf` middleware and embedded in the HTML page as a meta tag and a global `window.csrfToken` variable.

All `fetch` requests from the admin dashboard include the token in the `CSRF-Token` header. Requests without a valid token are rejected with a `403` status. The `/csrf-token` endpoint exists for browser flows and tests.

## Dashboard Rendering

Values rendered in the dashboard are escaped before being inserted into HTML or inline JavaScript to reduce XSS exposure.

## Double Anonymity

Client IP addresses are never logged or stored; only an irreversible SHA256 hash with a daily rotating salt is kept. Rate limiting uses the hashed IP.

## HTTPS

In production, the service should be deployed behind a reverse proxy with HTTPS enabled. Set `SESSION_SECRET` to a strong random value. Security headers (Helmet) and CORS are configured conditionally based on the `PUBLIC_URL` protocol.
